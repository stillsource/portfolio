# Research: Palettes CIELAB & Générateur de Contenu Astro

**Date**: 2026-04-08

## 1. Executive Summary
This research maps the current implementation of the `solenya-sync` Go tool and identifies the components related to image processing, metadata extraction, and Astro content generation. The project currently uses K-means clustering in Lab space for single-image palette generation and has a defined Zod schema for Astro content.

## 2. Technical Context
- **Existing Palette Logic**: Located in `src/scripts/go/solenya-sync/internal/processor/processor.go:95`. It uses the `github.com/lucasb-eyer/go-colorful` library to perform K-means clustering on a subsampled set of pixels (32x32) from a single image.
- **Astro Content Schema**: Defined in `src/types/content.ts`.
    - `exifSchema` (line 3): Includes `shutter`, `aperture`, `iso`, `body`, `lens`, and `focalLength`.
    - `rollSchema` (line 13): Includes `title`, `date`, `tags`, `poem`, `palette`, `dominantColor`, and an array of `images`.
- **Go Project Structure**: The sync tool is located in `src/scripts/go/solenya-sync/`. It currently consists of:
    - `cmd/sync/main.go`: Basic orchestration (currently only handles kDrive file listing).
    - `internal/kdrive/client.go`: Client for kDrive API.
    - `internal/processor/processor.go`: Image processing (resize, blurhash, palette, watermark).

## 3. Findings & Analysis
- **Library for EXIF**: `github.com/evanoberholster/imagemeta` provides high-performance metadata decoding.
    - `imagemeta.Decode` returns an `exif2.Exif` object.
    - Relevant tags for `exifSchema` mapping:
        - `exif2.ExposureTime` (0x829a) -> `shutter`
        - `exif2.FNumber` (0x829d) -> `aperture`
        - `exif2.ISOSpeedRatings` (0x8827) -> `iso`
        - `exif2.Model` (0x0110) -> `body`
        - `exif2.LensModel` (0xa434) -> `lens`
        - `exif2.FocalLength` (0x920a) -> `focalLength`
- **Palette Logic Detail**: The `GeneratePalette` function (`processor.go:95`) implements K-means clustering by:
    1. Subsampling the image.
    2. Initializing centroids.
    3. Iterating to minimize distance in CIELAB space (`p.DistanceLab(c)`).
- **Astro Frontmatter Format**: Existing files in `src/content/rolls/` (e.g., `nuit-a-tokyo.md`) show a YAML frontmatter structure that includes roll-level metadata and an array of image-level metadata (including `exif` and `palette`).

## 4. Technical Constraints
- **Library Availability**: `github.com/evanoberholster/imagemeta` is not yet in `go.mod`.
- **Astro Schema Strictness**: The `rollSchema` in `src/types/content.ts` is enforced by Astro. Any generated Markdown must strictly adhere to this structure for the site to build correctly.
- **Processor Dependency**: `GeneratePalette` is currently a private method of the `Processor` struct in `internal/processor/processor.go`.

## 5. Architectural Patterns
- **Package Organization**: The project follows an `internal/` directory pattern for separating concerns (e.g., `internal/kdrive`, `internal/processor`).
- **Dependency Management**: Uses Go modules (`go.mod`, `go.sum`).
- **Astro Integration**: Content is managed via Markdown files in `src/content/rolls/` which are loaded using the `glob` loader in `src/content.config.ts`.
